一、平台定位

智能数据操作系统

以本体（Ontology）为核心数据模型，融合 Palantir Foundry 的本体驱动架构与 AI Agent 的自主采集能力。

  • 自然语言描述需求 → Agent 自动获取数据
  • 统一语义模型理解与展示全部数据
  • Action / Function 在数据之上构建业务操作
  • 为上层 AI 应用与智能体平台提供标准化数据底座
定位公式

Foundry 的本体能力 + AI Agent 的自主性 = 数据不再需要人搬运。

目标：从「人找数据、人搬数据」转向「意图驱动、语义一致、可操作的统一层」。

二、核心概念 · 本体

数据的骨架，不是数据本身

定义 Object Type、属性、Link；描述「世界长什么样」。支持人工预定义与Agent 推断混合：先推断草案 → 人工审核固化。

方式 场景
人工预定义领域模型清晰、需严格管控
Agent 推断探索性分析、快速原型
本体 · Schema 演进

变更策略（版本可追溯）

变更处理
新增属性历史对象为 null，不破坏存量
属性重命名别名映射，旧名仍有效，后台迁移
删除属性deprecated + 归档，不立刻删数据
新增 Link对历史无影响
改属性类型需人工确认 + 迁移脚本

跨站点实体合并与去重

Action & Function

Action — 操作手

定义「能对数据做什么」：创建 Link、生成报告、触发刷新等。支持 UI、定时任务、自然语言触发。

Function — 计算器

只读计算：市场集中度（HHI）、关联路径、相似实体推荐等，返回派生结果，不改库存数据。

赋能方式

统一底座 · 即插即用 · 降本 · 联动 · AI

  • 统一数据底座：多上层平台共享同一 Object 定义，消除「同实体多 Schema」
  • API 能力：查询对象、触发 Action、执行 Function、派遣 Agent、路径查询、变更订阅
  • 成本：数据层开发常见占总工作量 40–60%；接入后有望降至约 10–15%
  • 跨平台联动：基于同一本体的变更事件，减少点对点定制对接
  • AI 工具集：search / detail / path / collect / action / function；支持 MCP
