# 数据自治平台（Ontology Platform）产品报告

> 以本体为骨架，以智能体为引擎，让数据自己生长、自己组织、自己服务。

---

## 一、平台定位

数据自治平台是一个 **以本体（Ontology）为核心数据模型** 的智能数据操作系统。它融合了 Palantir Foundry 的本体驱动架构与 AI Agent 的自主数据采集能力，让组织能够：

- 用自然语言描述需要什么数据，Agent 自动去获取
- 用统一的语义模型理解和展示所有数据
- 用 Action 和 Function 在数据之上构建业务操作
- 为上层 AI 应用和智能体平台提供标准化数据底座

**一句话定位：** Foundry 的本体能力 + AI Agent 的自主性 = 数据不再需要人搬运。

---

## 二、核心概念

### 2.1 本体（Ontology）— 数据的骨架

本体定义了"世界长什么样"。它不是数据本身，而是数据的结构和语义。

```
本体定义示例：

Object Type: 公司
├── 属性: 名称(string), 行业(enum), 营收(number), 成立时间(date)
├── Link: 竞争对手 → 公司
├── Link: 创始人 → 人物
└── Link: 所属行业 → 行业分类

Object Type: 人物
├── 属性: 姓名(string), 职位(string), 教育背景(string)
├── Link: 任职于 → 公司
└── Link: 投资了 → 公司
```

**平台支持两种本体构建方式：**

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 人工预定义 | 领域专家在平台上定义 Object Type 和 Link Type | 有明确业务模型时 |
| Agent 推断 | Agent 爬取数据后自动识别实体和关系，建议本体结构 | 探索性分析、快速原型 |

两种方式可以混合使用：先让 Agent 推断一个初始本体，人工审核修正后固化。

**本体 Schema 演进策略：**

本体驱动架构的关键挑战是：当 Object Type 定义改变时，历史数据如何处理。平台采用以下策略：

| 变更类型 | 处理方式 |
|---------|---------|
| 新增属性 | 历史对象该属性值为 null，不破坏现有数据 |
| 属性重命名 | 记录别名映射，旧属性名继续有效，后台迁移 |
| 删除属性 | 标记为 deprecated，数据归档保留，不立即删除 |
| 新增 Link Type | 无影响，历史对象无该关系即可 |
| 修改属性类型 | 强制类型变更需人工确认 + 迁移脚本，不自动执行 |

所有 Schema 变更均记录版本历史，可查看任意历史版本的本体定义。

### 2.2 智能体（Agent）— 数据的猎手

Agent 是平台的自主数据采集引擎。用户只需描述意图，Agent 自动完成数据获取。

**Agent 工作流程：**

```
用户意图                    Agent 执行                     平台结果
───────                    ────────                      ────────
"帮我收集中国                1. 规划爬取策略                 本体中新增:
 Top 50 新能源               2. 识别目标网站                 50个"公司"对象
 公司的信息"                  3. 爬取并提取结构化数据          属性全部填充
                             4. 推断/匹配本体结构             关系自动建立
                             5. 数据清洗去重
                             6. 写入本体
```

**Agent 的核心能力：**

- **通用爬取** — 给定 URL 或搜索意图，自动提取结构化数据
- **本体对齐** — 将非结构化数据映射到已有本体，或建议新的本体结构
- **增量更新** — 定期重访数据源，发现变化自动更新
- **多源融合** — 同一实体来自多个网站的数据自动合并去重

### 2.3 Action — 数据的操作手

Action 定义了"可以对数据做什么"。它让本体不仅是只读的展示层，而是可操作的业务层。

```
Action 示例：

Action: 标记竞争关系
├── 输入: 公司A, 公司B
├── 逻辑: 在A和B之间创建"竞争对手"Link
├── 副作用: 通知关注这两家公司的用户
└── 触发方式: UI按钮 / 自然语言("把华为和爱立信标记为竞争对手")

Action: 生成行业报告
├── 输入: 行业分类
├── 逻辑: 聚合该行业所有公司数据，调用LLM生成分析报告
├── 副作用: 生成PDF存入文档库
└── 触发方式: UI按钮 / 自然语言("给我出一份新能源行业报告")

Action: 触发数据刷新
├── 输入: Object Type 或具体对象
├── 逻辑: 派遣Agent重新爬取该对象的最新数据
├── 副作用: 更新本体中的属性值，记录变更历史
└── 触发方式: UI按钮 / 定时任务 / 自然语言
```

### 2.4 Function — 数据的计算器

Function 是纯计算逻辑，不修改数据，只返回结果。用于派生指标、搜索、聚合等。

```
Function 示例：

Function: 计算市场集中度
├── 输入: 行业分类
├── 逻辑: 取该行业Top N公司营收，计算HHI指数
└── 返回: number

Function: 查找关联路径
├── 输入: 实体A, 实体B
├── 逻辑: 在关系图中查找A到B的所有路径（≤3跳）
└── 返回: Path[]

Function: 相似实体推荐
├── 输入: 任意实体
├── 逻辑: 基于属性和关系拓扑计算相似度
└── 返回: Entity[]（按相似度排序）
```

## 三、功能全景

### 3.1 数据采集层

| 功能 | 描述 |
|------|------|
| 意图驱动采集 | 自然语言描述需求，Agent自动规划并执行爬取 |
| URL定向采集 | 指定网站/页面，自动提取结构化数据 |
| API对接 | 支持接入第三方API作为数据源 |
| 文件导入 | 支持CSV、Excel、JSON、PDF等文件导入 |
| 定时刷新 | 数据源定期重访，保持数据新鲜度 |
| 采集监控 | 查看Agent采集状态、成功率、数据质量评分 |

### 3.2 本体管理层

| 功能 | 描述 |
|------|------|
| 可视化本体编辑器 | 拖拽式定义Object Type、属性、Link Type |
| Agent辅助建模 | Agent根据采集数据建议本体结构，人工一键确认/修正 |
| 本体版本管理 | 本体结构变更有版本记录，可回溯 |
| Schema演进 | 新增属性/类型时自动兼容历史数据（见第二节演进策略） |
| 本体模板库 | 预置行业通用本体（公司、人物、事件、产品…），开箱即用 |
| 数据血缘追踪 | 每个属性值记录来源URL、抓取时间、提取置信度，支持全链路溯源 |

### 3.3 数据展示层

| 功能 | 描述 |
|------|------|
| 对象详情页 | 以本体结构自动生成实体详情页（属性+关联关系） |
| 关系图谱 | 可视化探索实体间的关系网络，支持多跳展开 |
| 列表/表格视图 | 按Object Type浏览，支持筛选、排序、分组 |
| 时间线视图 | 以时间维度展示实体的变化历程 |
| 仪表盘 | 基于Function的聚合指标构建可视化面板 |
| 自然语言查询 | "营收超过1000亿的通信公司有哪些？"直接返回结果 |

### 3.4 业务操作层

| 功能 | 描述 |
|------|------|
| Action定义与执行 | 在本体上定义可执行的业务操作 |
| 自然语言触发 | 用对话方式触发Action，无需找按钮 |
| 批量操作 | 对筛选后的对象批量执行Action |
| 操作审计 | 所有Action执行记录可追溯 |
| 权限控制 | 按角色/用户控制谁能执行哪些Action |
| 工作流编排 | 将多个Action串联为自动化流程 |

### 3.5 数据治理层

| 功能 | 描述 |
|------|------|
| 字段级数据血缘 | 每个属性值记录完整来源链路，知道数据从哪来、经过什么处理 |
| 可信度评分 | 每条数据有置信度标注，来源越权威、确认次数越多分数越高 |
| 人工审核队列 | 争议数据和低置信度数据进入审核队列，不自动入库 |
| 变更历史 | 对象属性的每次变更都有时间戳记录，可查看历史快照 |
| 数据分级标注 | 支持对数据标记敏感度等级（公开/内部/受限），配合权限控制 |

### 4.2 对上层平台的赋能方式

#### 赋能方式一：统一数据底座

> 上层平台不再各自建数据管道，而是共享一个语义一致的数据层。

```
❌ 没有本平台时：
    行业研究平台 → 自己爬数据 → 自己的"公司"表 → 字段定义各异
    风险监控平台 → 自己爬数据 → 自己的"公司"表 → 字段定义各异
    （同一家公司，两个平台的数据不一致）

✅ 有本平台后：
    数据自治平台 → 统一的"公司"Object Type → 统一的数据
    行业研究平台 → 读取 → 一致的数据
    风险监控平台 → 读取 → 一致的数据
```

#### 赋能方式二：即插即用的数据能力

上层平台通过 API 调用本平台能力，不需要自己实现：

```
上层平台可以直接调用的能力：

GET  /ontology/objects?type=公司&filter=行业=新能源    ← 查询数据
POST /ontology/actions/trigger                         ← 触发Action
GET  /ontology/functions/execute?name=市场集中度        ← 执行Function
POST /ontology/agent/collect                           ← 派遣Agent采集新数据
GET  /ontology/graph/paths?from=A&to=B                 ← 关系路径查询
WS   /ontology/subscribe?type=公司                     ← 数据变更订阅
```

#### 赋能方式三：降低上层平台的构建成本

| 上层平台自己做 | 用本平台后 |
|--------------|-----------|
| 写爬虫、维护反爬策略 | 一句话让Agent去采集 |
| 设计数据库Schema | 复用或扩展已有本体 |
| 开发数据清洗流程 | Agent自动清洗去重 |
| 构建关系图谱能力 | 直接调用图谱API |
| 实现数据更新机制 | 定时刷新已内置 |

**量化估算：** 一个上层业务平台的数据层开发工作量通常占总量的 40-60%。接入本平台后，这部分可以缩减到 10-15%（仅剩对接和定制）。

#### 赋能方式四：跨平台数据联动

```
场景：供应链风险预警

1. 风险监控平台发现"某公司"被列入制裁名单
2. 触发Action → 更新该公司的风险标签
3. 数据变更事件自动推送给供应链管理平台
4. 供应链平台自动识别受影响的供应链路径
5. 触发通知Action → 告警相关负责人

整个链路不需要任何平台之间做专门对接——
它们都基于同一个本体，数据变更天然互通。
```

#### 赋能方式五：为 AI 智能体平台提供数据支撑

> AI Agent 需要工具来操作世界；本平台将本体数据能力封装为 Agent 可直接调用的工具集。

```
AI 智能体平台可以使用本平台提供的工具：

Tool: search_entities        ← 按条件检索实体
Tool: get_entity_detail      ← 获取实体完整信息（属性+关系）
Tool: find_relationship_path ← 查找两个实体间的关系路径
Tool: collect_data           ← 派遣采集Agent获取新数据
Tool: execute_action         ← 触发平台内定义的Action
Tool: run_function           ← 执行计算Function，获取派生指标
```

**协议支持：**

| 接入方式 | 说明 | 适用场景 |
|---------|------|---------|
| REST API | 标准HTTP接口，通用性最强 | 业务平台集成 |
| MCP（Model Context Protocol） | 标准化AI工具协议，Claude/GPT等直接调用 | AI智能体平台 |
| Python SDK | 封装常用操作，减少样板代码 | 开发者快速集成 |
| WebSocket 订阅 | 数据变更实时推送 | 需要响应式更新的应用 |

这意味着外部 AI Agent 可以通过 MCP 协议，像使用工具一样直接查询和操作本平台的本体数据，无需了解底层数据结构。

## 五、典型使用场景

### 场景一：竞争情报分析

```
用户："帮我持续追踪国内Top 20 自动驾驶公司的动态"

平台执行：
1. Agent 搜索并识别Top 20公司
2. 自动创建/匹配"公司"对象，填充属性
3. 建立公司间的竞争、投资、合作关系
4. 设置定时任务，每日刷新动态
5. 新闻、融资、人事变动自动关联到对应公司

用户得到：
- 20家公司的完整画像
- 可视化关系图谱
- 每日变动推送
- 自然语言查询："最近一个月融资额最高的是哪家？"
```

### 场景二：供应链映射

```
用户："帮我梳理苹果公司在中国的供应链"

平台执行：
1. Agent 从公开信息爬取苹果供应商名单
2. 建立"供应商" Object Type 和"供货"关系
3. 继续爬取每个供应商的子供应商（多层展开）
4. 标注每个节点的地理位置、主营业务、产能

用户得到：
- 多层供应链关系图谱
- 按地域/品类的供应商分布
- Action: "如果某个供应商中断，影响哪些产品线？"（路径分析Function）
```

### 场景三：学术研究网络

```
用户："帮我分析 AI 领域 2024-2025 年高引论文的作者关系网络"

平台执行：
1. Agent 从学术网站爬取高引论文数据
2. 创建"论文"、"作者"、"机构"Object Type
3. 建立合著、引用、所属机构等关系
4. 计算Function: 影响力指数、合作紧密度

用户得到：
- 作者合作关系图谱
- 核心研究团体识别
- 跨机构合作趋势
- "谁是连接 Stanford 和 清华 AI 研究的桥梁人物？"
```

## 七、与 Palantir Foundry 的差异化总结

| 维度 | Palantir Foundry | 数据自治平台 |
|------|-----------------|-------------|
| 数据获取 | 企业内部数据对接为主，需要Connector开发 | AI Agent自主采集 + 传统对接 |
| 建模方式 | 人工定义Ontology | 人工 + Agent推断建议 |
| 操作方式 | 开发者写代码定义Action | 支持自然语言触发 |
| 用户门槛 | 高，需要专业培训 | 低，自然语言交互 |
| 数据范围 | 偏企业内部数据 | 内外部数据并重 |
| 部署方式 | 重量级，企业级部署 | 可轻量启动，渐进扩展（本地/云均可） |
| 目标用户 | 大型企业/政府 | 研究机构、中小企业、数据团队 |
| AI 智能体集成 | 无原生支持 | MCP协议 + Tool接口，AI Agent可直接调用 |
| 数据治理/合规 | 企业级细粒度权限、数据分级 | 轻量治理：字段级溯源 + 可信度评分 + 人工审核队列 |
| 核心壁垒 | 平台成熟度和政府关系 | AI驱动的数据自治能力 + AI原生集成接口 |

