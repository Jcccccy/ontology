# 数据自治平台 MVP 实施报告

> 1人 + AI 开发，以公开政治人物关系网络为切入场景，快速验证核心价值。

---

## 一、MVP 范围界定

### 做什么 / 不做什么

| 做 | 不做 |
|----|------|
| 单用户本地部署 | 多用户、权限系统 |
| Agent 自动爬取（**核心亮点，不简化**） | 复杂工作流编排 |
| Agent 推断本体结构 | 可视化拖拽本体编辑器 |
| 本体驱动的数据展示 | 仪表盘/报表 |
| Action & Function 基础能力 | 批量操作、操作审计 |
| 自然语言查询 | 时间线视图 |
| 关系图谱可视化 | 本体版本管理 |

### MVP 验证场景：公开政治人物关系网络（以特朗普核心圈为样例）

```
用户（懂政治分析，不懂技术）的典型操作流程：

1. 输入："帮我收集特朗普核心圈子的人物信息和他们之间的关系"

2. Agent 自动执行：
   → 搜索并爬取特朗普内阁成员、顾问、家族成员信息
   → 提取结构化数据（姓名、职位、背景、政治立场...）
   → 推断本体结构：人物、组织、事件、政策
   → 建立关系：任命、家族、商业伙伴、政治盟友/对手
   → 返回建议："我发现了以下实体类型和关系，是否确认？"

3. 用户确认后，平台展示：
   → 人物列表（属性卡片）
   → 关系图谱（谁和谁有什么关系）
   → 自然语言查询："谁是特朗普的经济政策核心顾问？"

4. 用户继续深挖：
   → "帮我把马斯克和特朗普之间的关系链梳理出来"
   → "最近一个月特朗普提名了哪些新人选？"
   → Action: "把这个人标记为鹰派" / "生成这组人物的关系简报"
```

**场景边界：**
- 首批数据源限定为 Wikipedia、官方机构页面、公开人物介绍页、少量白名单媒体
- MVP 只处理公开可验证信息，不做主观政治倾向判断
- 争议性关系默认进入人工确认队列，不自动入库

**为什么选这个场景：**
- 数据公开，适合快速验证 Agent → 本体 → 图谱 的核心链路
- 关系网络复杂，能充分展示本体和图谱能力
- 业务用户有真实的关系梳理和人物画像需求
- 演示效果强，便于内部汇报和外部沟通

**MVP 技术选型理由：**

| 选择 | 理由 |
|------|------|
| Python + FastAPI | 你和AI开发效率最高，LLM生态最好 |
| Next.js | 前后端分离，AI辅助生成UI效率高 |
| PostgreSQL 单库 | MVP不需要多数据库运维负担，JSONB足够灵活 |
| Redis + Celery | Agent采集是耗时异步任务，不能阻塞HTTP请求；Celery成熟、文档多、AI生成代码质量高 |
| Claude API（主） | Agent大脑，负责爬取规划、本体推断、自然语言理解；优先Claude，备用GPT-4o |
| D3.js / force-graph | 关系图谱可视化，轻量够用 |
| Playwright（按需） | 遇到 JS 渲染页面时启用；Wikipedia等静态页面用 httpx + BeautifulSoup 即可，不默认引入 |

**关于网页提取的分级策略：**

```
目标网页
   │
   ├── 静态HTML（Wikipedia、大多数官方页面）
   │       → httpx + BeautifulSoup → LLM提取
   │
   └── JS渲染页面（媒体网站、部分政府网站）
           → Playwright Headless → 等待渲染 → LLM提取
           （MVP阶段按需引入，白名单中标注哪些URL需要此路径）
```

**何时从 PostgreSQL 迁移到 Neo4j：**

MVP 阶段用 PostgreSQL 递归 CTE 处理图查询。以下任一条件触发时，评估引入 Neo4j：
- 单次图遍历（≤3跳）查询超过 500ms，且调优后无法改善
- 节点总数超过 100 万，关系总数超过 1000 万
- 需要支持 ≥4跳 的路径查询，或复杂的图算法（PageRank、社区发现等）

## 三、自然语言查询的技术路径

MVP 的自然语言查询需要回答两类问题：

```
查询类型一：结构化过滤
  "共和党的内阁成员有哪些？"
  → LLM 将自然语言转换为内部过滤条件
  → Ontology Engine 执行属性过滤查询
  → 适合: 属性筛选、聚合统计

查询类型二：语义相似
  "和特朗普理念相近的人有哪些？"
  → 对象属性文本向量化（pgvector 扩展）
  → 向量相似检索返回候选
  → 适合: 模糊匹配、概念相似

查询类型三：图关系
  "谁和马斯克有直接关系？"
  → LLM 识别为图遍历查询
  → Ontology Engine 执行关系图遍历
  → 适合: 关系链路、路径查找
```

MVP 阶段实现一和三，二作为可选增强（pgvector 安装简单，视第三阶段进度决定是否加入）。

## 四、可信机制设计（MVP 必做）

MVP 不是"自动抓完直接信"，而是"自动提取 + 可追溯 + 可审核"。

### 4.1 字段级可信信息

每个实体属性、每条关系都尽量保留下列元数据：

- 来源 URL
- 来源站点名称
- 抓取时间
- 提取置信度
- 是否人工确认
- 原始片段或摘要

### 4.2 入库分层

```
原始网页/文档
   ↓
raw_sources
   ↓
LLM提取后的候选实体/关系
   ↓
entity_candidates
   ↓
人工确认/修正
   ↓
objects / links
```

### 4.3 冲突处理规则

- 同一字段多个来源不一致时，不自动覆盖高可信已确认数据
- 高争议字段进入 `review_queue`
- 对外展示时优先显示"当前采用值 + 来源"
- 对象详情页提供"查看来源"入口，增强用户信任

### 4.4 审核队列 UI 设计（第一阶段必做）

审核队列是用户每天使用的核心界面，需要足够高效：

```
审核队列页面布局：

┌─────────────────────────────────────────────────┐
│  待审核：23条   已通过：142条   已拒绝：8条         │
├─────────────────────────────────────────────────┤
│  [筛选: 全部 / 低置信度 / 冲突 / 新实体]            │
├──────────────────────────────┬──────────────────┤
│  候选实体/关系列表             │  详情面板          │
│                              │                  │
│  ○ 马斯克 · 人物 · 置信0.82   │  姓名: 埃隆·马斯克  │
│    来源: Wikipedia            │  来源A: "CEO of  │
│    ⚠ 与已有记录属性冲突        │    Tesla"        │
│                              │  来源B: "CEO of  │
│  ○ 任命关系 · 置信0.71        │    SpaceX"       │
│    马斯克 → DOGE              │                  │
│    来源: 官方公告              │  ── 冲突字段 ──   │
│                              │  职位:            │
│  ...                         │  [已有] "Tesla CEO"│
│                              │  [新值] "DOGE负责人"│
│                              │  ○ 保留已有        │
│                              │  ○ 采用新值        │
│                              │  ○ 两者都保留      │
│                              │                  │
│                              │  [✓ 确认] [✗ 拒绝]│
└──────────────────────────────┴──────────────────┘
```

关键交互原则：
- 每条候选项一键确认/拒绝，不要让用户填表
- 冲突字段高亮对比，用户只需选择而非手动输入
- 支持键盘快捷键（J/K 导航，Y/N 确认拒绝）

## 五、Prompt 管理策略

Agent 的核心能力由 Prompt 驱动，需要有可维护的管理方式。

### 5.1 Prompt 目录结构

```
prompts/
├── agent/
│   ├── planner_v1.yaml        ← 爬取规划器
│   ├── extractor_v1.yaml      ← 实体提取器
│   └── ontology_infer_v1.yaml ← 本体推断器
├── nl_interface/
│   ├── query_parser_v1.yaml   ← 自然语言查询解析
│   └── result_summary_v1.yaml ← 结果摘要生成
└── action/
    └── report_gen_v1.yaml     ← 简报生成
```

每个 YAML 文件结构：

```yaml
name: entity_extractor
version: v1
description: 从网页内容中提取结构化实体信息
model: claude-sonnet-4-6
temperature: 0.1
system: |
  你是一个结构化信息提取专家...
user_template: |
  网页内容：{{content}}
  目标本体类型：{{object_type}}
  请提取...
few_shots:
  - input: "..."
    output: "..."
```

### 5.2 Prompt 迭代原则

- 每次修改 Prompt 递增版本号（v1 → v2），不覆盖旧版本
- 用验证集（见第七节）对比新旧版本效果后再切换
- Agent 代码引用 Prompt 时指定版本号，便于回滚

## 六、分阶段实施计划

### 第一阶段：地基（第1-2周）

**目标：能存数据、能看数据、能审核数据**

```
交付物：
├── PostgreSQL Schema 设计并建表
├── Ontology Engine 核心 API
│   ├── POST /types           创建 Object Type / Link Type
│   ├── POST /objects         创建对象
│   ├── POST /links           创建关系
│   ├── GET  /objects?type=   按类型查询对象
│   └── GET  /objects/:id     获取对象详情（含关联关系）
├── Redis + Celery 基础配置（为第二阶段做准备）
├── 前端骨架
│   ├── 对象列表页（按类型浏览）
│   ├── 对象详情页（属性 + 关联关系卡片 + 来源信息）
│   └── 审核队列页（候选列表 + 详情面板 + 冲突对比 + 一键确认/拒绝）
└── 手动导入"特朗普核心圈"种子数据（10-20人）验证流程
```

**验收标准：** 能手动创建"人物"类型，录入特朗普、马斯克等人物，建立关系，在前端浏览，字段来源可见；审核队列能对候选数据做确认/拒绝操作。

---

### 第二阶段：Agent 大脑（第3-5周）⭐ 核心亮点

**目标：Agent 能自动采集候选数据，并在人工确认后写入本体**

```
交付物：
├── Agent Engine
│   ├── 爬取规划器（LLM 根据用户意图生成采集计划）
│   ├── 白名单网页提取器（URL → 结构化数据）
│   │   ├── 静态页面: httpx + BeautifulSoup + LLM提取
│   │   ├── JS渲染页面: Playwright（白名单中标注需要此路径的站点）
│   │   └── 使用 LLM 从网页内容中提取实体和属性
│   ├── 本体建议器
│   │   ├── 根据提取数据推断 Object Type 和属性
│   │   ├── 推断实体间关系类型
│   │   └── 生成建议，等待用户确认
│   ├── 数据清洗 & 去重
│   │   ├── 同一人物多个来源的数据合并
│   │   └── 冲突检测（不同来源数据矛盾时提示用户）
│   └── 候选写入管道（清洗后数据 → 候选区 → 人工确认 → Ontology Engine）
│
├── Agent API
│   ├── POST /agent/collect    {"intent": "收集特朗普内阁成员信息"}
│   ├── GET  /agent/tasks/:id  查看采集进度
│   └── POST /agent/confirm    确认Agent的本体推断建议
│
└── 前端：Agent 交互面板
    ├── 输入意图
    ├── 查看采集进度（实时流式输出Agent在做什么）
    ├── 审核Agent推断的本体结构（确认/修正）
    └── 查看采集结果摘要和来源
```

**Agent 工作流详细设计：**

```
用户输入: "帮我收集特朗普第二任期的内阁成员"
                    │
                    ▼
        ┌─── 规划器 (LLM) ───┐
        │                    │
        │  分析意图，生成计划: │
        │  1. 搜索内阁成员名单 │
        │  2. 逐人爬取详细信息 │
        │  3. 识别人物间关系   │
        └──────────┬──────────┘
                   │
                   ▼
        ┌─── 提取器 (LLM + 白名单采集) ───┐
        │                                │
        │  对每个URL:                     │
        │  1. 判断是否需要Playwright渲染   │
        │  2. 获取页面内容                │
        │  3. LLM提取结构化信息           │
        │  4. 输出: {name, role,          │
        │     background, ...}           │
        └──────────┬──────────────────────┘
                   │
                   ▼
        ┌─── 推断器 (LLM) ───┐
        │                    │
        │  分析提取的数据:     │
        │  建议 Object Types: │
        │  · 人物(姓名,职位,  │
        │    党派,背景...)    │
        │  · 政府部门(名称,   │
        │    职能...)         │
        │  建议 Link Types:   │
        │  · 任命(人物→部门)  │
        │  · 政治盟友(人物↔人)│
        └──────────┬──────────┘
                   │
                   ▼
               用户审核确认
                   │
                   ▼
        ┌─── 候选写入管道 ───┐
        │                   │
        │  清洗 → 去重       │
        │  → 候选区          │
        │  → 审核通过入库    │
        └───────────────────┘
```

**验收标准：** 输入"帮我收集特朗普内阁成员"，Agent 能从白名单数据源采集候选数据、推断本体建议、用户确认后自动入库，前端可浏览。

### 第三阶段：能看关系、能问问题（第6-7周）

**目标：关系图谱可视化 + 自然语言查询**

```
交付物：
├── 关系图谱组件
│   ├── 以选定人物为中心展开关系网络
│   ├── 支持点击节点展开下一层关系
│   ├── 关系类型用颜色/标签区分
│   └── 支持缩放、拖拽、搜索定位
│
├── 自然语言查询
│   ├── NL Interface 层（LLM 将自然语言转为内部查询）
│   ├── 支持属性过滤："共和党的内阁成员有哪些？"
│   ├── 支持一跳/两跳关系查询："谁和马斯克有直接关系？"
│   ├── 支持基础路径查询："特朗普和某人之间有什么关联？"
│   └── 暂不支持开放式复杂分析问答
│
└── 对话界面增强
    ├── 查询结果内嵌对象卡片（可点击跳转）
    └── 查询结果内嵌迷你图谱
```

**验收标准：** 打开特朗普的关系图谱能看到核心人物网络；问属性过滤和一跳/两跳关系类问题能返回正确答案。

### 第四阶段：能操作（第8-9周）

**目标：Action & Function 让平台不只是展示，还能做事**

```
交付物：
├── Action 系统
│   ├── 内置 Action
│   │   ├── 标记/打标签（给人物标记"鹰派""鸽派"）
│   │   └── 生成关系简报（选一组人物，LLM生成分析文档）
│   ├── 自然语言触发："把蓬佩奥标记为对华鹰派"
```

## 八、技术风险 & 应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| Agent 爬取被反爬封锁 | 数据采集失败 | 优先爬取 Wikipedia/公开政府网站等友好源；加请求间隔；准备备用数据源 |
| LLM 提取准确率不够 | 数据质量差 | 设计人工审核环节；用few-shot prompt提高准确率；保留原始数据可回溯 |
| Playwright 引入增加复杂度 | 开发延期 | MVP 阶段优先覆盖静态页面；遇到 JS 渲染再按需引入，不提前引入 |
| 场景数据争议大 | 演示结果不被信任 | 先限定白名单数据源；争议关系默认不自动入库；展示来源和确认状态 |
| 关系图谱前端性能 | 节点多时卡顿 | MVP限制单次展示≤200节点；分层加载 |
| 一人开发进度风险 | 延期 | 严格MVP范围，砍功能不砍质量；AI最大化辅助编码 |

## 九、评估指标与回退方案

### 9.1 MVP 成功指标

| 指标 | MVP 目标 |
|------|----------|
| 单次采集任务成功率 | ≥ 70% |
| 人物基础属性提取准确率 | ≥ 80% |
| 关系识别准确率 | ≥ 70% |
| 候选数据人工确认通过率 | ≥ 60% |
| 从输入意图到首次可浏览结果 | ≤ 3 分钟 |
| 图谱首屏加载时间 | ≤ 2 秒（100节点内） |

### 9.2 验证集

- 预先准备 20-30 个标准人物样本
- 为每个人物准备基准字段：姓名、职位、所属组织、关键关系
- 每周对 Agent 提取结果做一次抽样评估，对比新旧 Prompt 版本效果，持续迭代

### 9.3 回退方案

如果"用户一句话 → Agent 全自动采集"在第5周前不稳定，则降级为：

1. 用户输入意图
2. 系统返回建议数据源 URL 列表
3. 用户确认或补充 URL
4. Agent 对已确认 URL 做提取
5. 用户审核后入库

这样可以保住演示主线，同时降低通用搜索和抓取的不确定性。

## 十、开发策略：1人 + AI 效率最大化

```
你（开发者）负责：              AI 辅助：
├── 架构决策                    ├── 生成 CRUD 代码和 API
├── 核心 Agent 逻辑设计         ├── 生成前端组件
├── Prompt Engineering          ├── 生成数据库 Schema 和迁移
├── 数据质量把控                ├── 写测试
├── 产品体验判断                ├── 调试和重构
└── 用户反馈收集                └── 文档生成
```

**原则：**
- 每个阶段结束时都要有可运行的产品，而不是半成品
- 宁可功能少但能用，不要功能多但半残
- Agent 是核心卖点，这部分的 Prompt 和流程要反复打磨
- Prompt 版本化管理，每次迭代保留历史版本，用验证集量化效果
