# 前端 Demo 补全任务书（供智能体生成界面）

> **依据**：`docs/product-report.md` 功能全景与核心概念。  
> **现状基线**：`docs/frontend-prototype-cn.html`（侧边三模块：智能体控制台、审核队列、关系图谱）。  
> **目标**：列出当前 Demo 与产品定义之间的缺口，并为后续智能体提供**可独立实现的页面级规格**，便于并行生成交互原型。

---

## 一、现有 Demo 已覆盖的能力（对照产品报告）

| 产品报告对应 | 现有实现要点 | 备注 |
|-------------|--------------|------|
| §2.2 Agent 工作流程、意图驱动 | 左侧对话 + 右侧任务流水线（规划→提取→推断→清洗） | 未单独展示「URL 定向」「增量」「多源融合」配置面 |
| §3.5 人工审核队列、冲突数据 | 审核列表 + 详情 + 冲突字段卡片 + 批准/拒绝 | 与实施报告一致，较完整 |
| §3.3 关系图谱 | D3 力导向图、多节点、关系标签、悬停高亮 | 缺「多跳展开」「从节点钻取详情」等交互说明级实现 |
| §2.1 / §3.3 对象与关系（隐含） | 审核详情里有人物/关系类型 | 无独立「对象详情页」「按类型的对象列表」 |

**结论**：Demo 强在 **Agent 叙事 + 审核 + 图谱壳子**；产品报告中的 **本体管理、完整数据展示层、采集监控、Action/Function、治理深度、集成与多场景** 大块未出现或仅局部触及。

## 二、缺口总览（按产品「功能全景」五层）

### 2.1 数据采集层（§3.1）— 基本是空

| 能力 | 现状 | 建议 Demo 形态 |
|------|------|----------------|
| 意图驱动采集 | 仅有对话入口 | 保留；可加「采集任务历史」列表（意图摘要、状态、时间） |
| URL 定向采集 | 无 | **新页面/抽屉**：粘贴 URL 列表、选择目标 Object Type、提交后进入任务队列 |
| API 对接 | 无 | **新页面**：简单「连接器」表单（Base URL、鉴权类型占位、同步频率） |
| 文件导入 | 无 | **新页面**：拖拽上传区 + 字段映射预览（CSV/Excel 示意） |
| 定时刷新 | 无 | **配置区**：Cron/周期选择 + 作用范围（某类型/某标签对象） |
| 采集监控 | 无 | **仪表盘页**：成功率、最近任务、质量评分趋势（可用静态假数据） |

### 2.2 本体管理层（§3.2）— 缺口最大

| 能力 | 现状 | 建议 Demo 形态 |
|------|------|----------------|
| 可视化本体编辑器 | 无（产品为拖拽式） | **新页面**：左侧类型列表，中间画布/树：Object Type + 属性表 + Link Type；MVP 可用「表单 + 简图」代替完整拖拽 |
| Agent 辅助建模 | 仅在 Agent 流里一句话带过 | **新页或模态**：「Agent 建议的本体变更 diff」— 新增属性/Link 勾选采纳 |
| 本体版本管理 / Schema 演进 | 无 | **侧栏或时间轴**：版本号、变更人、变更摘要；点开看 JSON/YAML diff（只读） |
| 本体模板库 | 无 | **模板市场栅格**：人物-公司-事件等卡片，一键「从模板创建工作区」 |
| 数据血缘（字段级） | 审核里有来源，无全链路 UI | **对象详情子模块**：每属性一行展示来源 URL、抓取时间、置信度、是否人工确认（与 §3.5 对齐） |

### 2.3 数据展示层（§3.3）— 部分缺失

| 能力 | 现状 | 建议 Demo 形态 |
|------|------|----------------|
| 对象详情页 | 无独立页 | **新页面**：面包屑 + 属性分组 + 关系统计 + 快捷跳图谱中心节点 |
| 列表/表格视图 | 无 | **新页面**：按 Object Type 筛选；列配置；排序/分组；行点击进入详情 |
| 时间线视图 | 无 | **详情页 Tab 或独立页**：属性变更/事件轴（时间戳 + 旧值→新值） |
| 仪表盘（Function 驱动） | 无 | **新页面**：2–4 个 KPI 卡 + 简单柱状/折线（数据标注为 Function 结果示例） |
| 自然语言查询 | 图谱搜索框占位 | **结果页**：问句 + 表格结果 + 可选「内嵌迷你图谱」（产品报告表述） |

### 2.4 业务操作层（§3.4）— 基本未展示

| 能力 | 现状 | 建议 Demo 形态 |
|------|------|----------------|
| Action 定义与执行 | 无 | **Action 目录**：卡片列出名称、输入参数、触发方式；详情侧栏展示「逻辑描述」（静态文案即可） |
| 自然语言触发 | 可与控制台合并 | 在对话中演示「触发某 Action」的伪回复（无需真实后端） |
| 批量操作 | 无 | **表格多选 + 批量执行 Action** 工具栏（禁用/确认对话框可 mock） |
| 操作审计 | 无 | **日志表**：时间、用户、Action、对象 ID、结果状态 |
| 权限控制 | 无 | **简易 RBAC 示意**：角色列表 + 每个角色可执行的 Action 勾选（只读 Demo） |
| 工作流编排 | 无 | **低复杂页**：A→B→C 节点连线示意图 + 说明「产品级为后续」（或只做静态 DAG 图） |

### 2.5 数据治理层（§3.5）— 仅有审核子集

| 能力 | 现状 | 建议 Demo 形态 |
|------|------|----------------|
| 字段级血缘 | 弱 | 见 2.2「对象详情」— 属性级来源链 |
| 可信度评分 | 弱 | **徽章/进度条**：全局分 + 分项因子说明（来源权重、确认次数） |
| 人工审核队列 | 有 | 可增加：队列维度的 SLA、优先级列 |
| 变更历史 | 无 | 与「时间线」共享数据模型展示 |
| 数据分级标注 | 无 | **对象/字段角标**：公开 / 内部 / 受限；筛选器 |

## 三、产品与「平台赋能」相关（§4）— Demo 可选一块

若需面向集成方或投资方：**单页「开发者门户」** 即够—

- REST 端点列表（表格）：路径、方法、一句话说明（与报告 §4.2 示例对齐）。
- 「MCP 工具」列表：`search_entities`、`execute_action` 等名称 + 参数摘要（静态）。
- **不必**实现真实鉴权；用 Tab 切换「REST / MCP / WebSocket 订阅说明」即可。

## 四、典型场景（§5）— 叙事扩展（可选）

当前 Demo 偏「政治人物关系」单一场景。可按产品报告拆 **3 个「剧本」面板**（或 3 个顶部 Tab）：

1. **竞争情报**：公司列表 + NL 问融资——表格 + 一句话结果。  
2. **供应链**：多层展开树或缩进列表 +「断供影响」按钮链到 Function 结果占位。  
3. **学术网络**：论文/作者/机构三类 Tab + 合作图谱占位。

**智能体任务**：先做 UI 壳与示例数据，不求真实算法。

## 五、推荐实施优先级（供排期）

| 优先级 | 模块 | 理由 |
|--------|------|------|
| P0 | 对象详情页 + 按类型列表/表格 | 产品核心路径「看数据」；与图谱、审核自然串联 |
| P0 | 自然语言查询结果页 | 产品差异化卖点；与 Agent 控制台形成闭环 |
| P1 | 本体管理（简版）+ Agent 建模建议 diff | 体现「本体驱动」；与报告 §2.1 对齐 |
| P1 | Action / Function 目录 + 一次 Function 结果展示 | 体现「不止是库，还能算、能做」 |
| P2 | 采集监控 + URL/文件导入 | 完善采集层故事，可静态仪表盘 |
| P2 | 治理：分级标签 + 审计日志 + 简易血缘 | 与报告 §3.5、§4 信任叙事一致 |
| P3 | 开发者/MCP 一页 + 多场景 Tab | 对外宣讲用 |

## 六、统一体验约束（给其他智能体的实现约定）

以下约定可减少多页面拼接时的割裂感：

1. **视觉**：延续 `frontend-prototype-cn.html` 的暗色底、网格/噪点、青/紫/琥珀强调色、`Noto Sans SC` + `JetBrains Mono`。  
2. **导航**：侧边栏或顶栏增加分组：**采集 / 本体 / 数据 / 操作 / 治理 / 系统**；当前「智能体 / 审核 / 图谱」归入「数据 + 治理」或「工作台」。  
3. **语言**：演示文案与标签以**中文**为主；API/MCP 标识符可保留英文。  
4. **交互深度**：允许**静态 JSON mock**、按钮 Toast、无后端；重点在**信息架构与关键控件**齐全。  
5. **无障碍**：新页面至少保证焦点可见、按钮有 `aria-label`（产品级可后续补强）。

对每一个新页面，建议按下列结构下达任务：

```markdown
- 入口：<从主导航哪一项进入>
- 主要组件：<列表>
- Mock 数据：<对象字段示例 1 个>
- 验收：<可点击路径 / 可见关键文案 / 移动端是否必须>
- 依赖：<是否依赖对象详情路由参数等>
```

## 八、与当前文件的衔接

- **不改** `frontend-prototype-cn.html` 也可并行产出 `frontend-demo-<module>.html` 多个文件，最后用主导航「跳转链接」聚合。  
- 若合并为单页应用，建议由人类或协调智能体统一 **路由与全局布局**，避免各自引入冲突的 CSS 变量名。

## 九、文档维护

- 当 `product-report.md` 增删功能模块时，同步更新**第二节表格**与**第五节优先级**。  
- 当某 Demo 已实现，在本 brief 对应行标记「✓ 已实现」及文件路径，避免多智能体重复劳动。
